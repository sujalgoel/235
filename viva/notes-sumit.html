04motivation
Motivation · why multimodal
You speak0:20

These are the three considerations that shaped our technical design.

Single models are not reliable across distributions. A detector that achieves 96 percent on FaceForensics may drop to 60 percent on a different dataset. We therefore needed more than one model per modality.

Ensembles mitigate this. Combining models that fail in different ways yields a system that is more reliable than any individual component.

Explainability was treated as a requirement. A flagged profile without justification cannot be acted upon.

05objectives
Objectives
You speak0:15

We set four objectives. Combine image, text and metadata signals. Make every prediction explainable. Calibrate the output to reflect actual confidence. Run the system locally without third party cloud calls.

We achieved all four objectives in version 1.0. The metadata module activates when an image carries EXIF data, which I will cover on slide 11.

06literature
Literature · what is established
You speak0:20

Three image pillars. XceptionNet from FaceForensics is the established baseline. EfficientNet B7 generalises better to unseen generators. CLIP provides zero shot sanity checking.

For text, the OpenAI RoBERTa detector and the Hello-SimpleAI ChatGPT detector together cover the major language model families.

The gap in the literature is honest cross dataset evaluation. We tested on a mixed set, closer to real deployment. You will see the cost on slide 16.

07solution
Solution · three lanes, one verdict
You speak0:20

Three independent analysis lanes. Each lane is an ensemble. The lanes are combined by a weighted fusion engine into a single trust score.

If a modality is missing or has low confidence, the weights renormalise automatically. The system degrades gracefully without misrepresenting its own confidence.

08architecture
Architecture · the central slide
You speak0:55

This is the system architecture, Figure 1.

The user input enters as a multipart POST containing the image and the bio text. Pydantic validates it before any model code runs.

The input is split into three streams. The image is routed to the image module. The bio is routed to the text module. The image EXIF is routed to the metadata module when present.

The image module contains three models. EfficientNet B7 at 50 percent weight. XceptionNet at 40 percent. CLIP at 10 percent.

The text module also contains three components. OpenAI's RoBERTa at 70 percent. The ChatGPT detector at 20 percent. A rule based check at 10 percent.

Each module returns a score and an explanation. Grad-CAM for image. SHAP and LIME for text.

All scores feed the weighted fusion engine. T equals 0.4 times the image score plus 0.3 times the text score plus 0.3 times the metadata score.

The output is mapped to one of four bands. Very low, low, moderate or high.

11metadata
Metadata module · EXIF forensics
You speak0:25

The metadata module extracts and scores image EXIF data. It examines four signals. The EXIF chain itself, since generated images rarely retain a complete chain. The camera signature, where make, model, lens and ISO consistency is verified. Editing software fingerprints, which pick up tags from Stable Diffusion, Photoshop AI and Midjourney exporters. And timestamp anomalies such as a Modify Date predating the Create Date.

These four signals fuse into a metadata score in the range 0 to 1. The EXIFTool dump on the right is an example, where missing Make, Model and Create Date combined with a Stable Diffusion software tag produces a metadata score of 0.12, indicating low trust.

When an uploaded image has its EXIF stripped, which is common on social platforms, the fusion engine adaptively renormalises and the trust score is computed from image and text alone. I will demonstrate that case on slide 12.

Equation 1 is weighted fusion. T equals 0.4 times image plus 0.3 times text plus 0.3 times metadata. Standard linear late fusion.

Equation 2 handles the missing modality case. When metadata is absent, the weights renormalise. Image becomes 0.571. Text becomes 0.429.

The pie on the right is Figure 3. It shows the four band ranges. Very low covers 0 to 0.30. Low covers 0.30 to 0.50. Moderate covers 0.50 to 0.70. High covers 0.70 to 1.0.

§ 2 · Q&A · panel questionsPrepared answers.

Q01The ensemble weights 0.50, 0.40 and 0.10. Did you derive these from a grid search, or are they informed estimates?

To be transparent, ma'am, they are informed by individual model accuracy on a validation split rather than by a full grid search. EfficientNet achieved the highest standalone accuracy on our set so it received the largest weight. XceptionNet was second. CLIP was third. We evaluated a few alternative combinations such as 0.4 0.4 0.2 and 0.6 0.3 0.1 but the difference in fused accuracy was within one percentage point, which is within the noise given our sample size. For a publication quality result I would conduct a proper grid search with cross validation. For this project the empirical weights were sufficient.

Q02What is the 95 percent confidence interval on your 67.8 percent image accuracy? How large is the test set?

The image test set contains 59 samples, 30 fake and 29 real. Using a normal approximation, the 95 percent confidence interval on 67.8 percent is approximately plus or minus 11.9 percentage points. So our true accuracy lies somewhere in the range of roughly 56 to 80 percent. We report the point estimate but the uncertainty is real. The AUC at 0.697 is more stable because it is computed from rankings rather than a single threshold. A larger test set would tighten both intervals.

Q03How do you confirm that your test set was not part of the pretraining data for EfficientNet or CLIP?

We cannot fully guarantee this, sir. EfficientNet was pretrained on ImageNet, which is publicly known. Our test images were drawn from a different distribution. The fakes come from thispersondoesnotexist dot com which generates a fresh face on every page load, so those are by definition not in any pretraining set. The reals come from recent Unsplash uploads which could overlap with the broader CLIP corpus. To rule this out completely, we would need to use only images created after CLIP's training cutoff and verify their provenance. We have noted this risk in the limitations chapter.

Q04You report accuracy but you also showed AUC. In a verification setting, which matters more?

AUC is more important for the deployed product, since operators can tune the threshold to their false positive tolerance. On the image side our AUC is 0.697, which means the ranking quality is real. On the text side our AUC is 0.500, which is honest about the fact that the text detector is at chance for ranking the two populations. Leading with accuracy on the slide is more intuitive for a non-ML audience. AUC is the more honest metric and that is a fair point ma'am.

Q05The text module has 94 percent recall but only 54 percent precision. Explain that asymmetry.

The text detector predicts AI for almost everything. It catches 47 of 50 AI bios, giving 94 percent recall on the AI class. But it also predicts AI for 40 of 50 human bios, which is why precision is only 54 percent. The underlying issue is that the score distributions for AI and human text overlap almost completely on our 100 sample set, AUC 0.500. The classifier has a bias toward the AI label. Retraining the head on recent paraphrased data is the correct fix.

Q06Calibration. Was your calibration set held out from both the training data and the test data?

Yes ma'am. The calibration set was a separate 30 percent split from the training data and never overlapped with the test set. Temperature scaling was fit on this calibration split and frozen before the final test evaluation. The reported calibration curve is therefore not optimistic.

Q07Why average softmax probabilities rather than logits, or learn a small MLP on top of the three image scores?

Averaging probabilities is simpler and more interpretable. Averaging logits would produce a sharper combined distribution, which can help in some cases but also amplifies confident wrong predictions. A learned MLP would likely perform better but we did not have sufficient labelled validation data to train one. For version 2.0 I would train a small MLP using held out cross validation.

Q08Late fusion. By averaging module scores, you discard any interaction between image and text. Was that a deliberate trade-off?

Yes ma'am, deliberate. Joint training of image and text requires paired labelled data at a scale we did not have. Late fusion allows us to use pretrained modules independently. The trade-off is that the system cannot model cases where a real image combined with a fake bio produces an informative interaction. A future version with paired training data would include a learned fusion head.

Q09Inference batching. If 50 requests arrive concurrently, does FastAPI queue them or process them in parallel? What is the throughput per instance?

FastAPI dispatches requests to a thread pool. The model inference holds the GIL during the forward pass so true parallelism is limited. A single instance sustains approximately four requests per second on CPU. For higher throughput we would run multiple worker processes behind a load balancer, or move to a batched inference server such as Triton.

Q10Cold start latency. Your median is 17 seconds. That is not acceptable for a live service.

The 17 second figure includes the first request after a fresh process start, where EfficientNet, RoBERTa and CLIP are loaded into memory. For production we would run the service as a persistent process behind a load balancer, with a health check that reports ready only after all models have loaded. The cold start is then paid once per instance lifetime, not per request. For a serverless deployment we would require pre-warming.

Q11The container size is approximately 4 gigabytes. Did you attempt weight quantisation or a slim base image?

We did not quantise the weights. EfficientNet B7 at FP16 would halve memory and reduce inference time. We have noted this as future scope. The base image is python slim. The bulk of the image is the model checkpoints themselves, which are baked in for reproducibility.

Q12CLIP at 10 percent weight. Is it pulling the fused score upward or downward on average? Could you drop it?

On our test set, removing CLIP shifted average accuracy by less than half a percentage point. It contributes meaningfully on a few specific samples such as recent diffusion outputs where the other two models are blind. For a slimmer production build I would consider dropping CLIP and accepting the small accuracy cost.

Q13What happens if a model crashes during a request?

Each module runs inside a try except block in the pipeline. A failed module returns None. The fusion engine treats it as a missing modality and renormalises the remaining weights. The client receives a partial result with an explicit indicator that one module failed. Never a 500 error.

Q14Your numbers do not match the headline figures of 98 percent.

The numbers match the README of our project. Our README states our measured accuracy on the test set we ran, which is what slide 16 and slide 17 show. The 98 percent figure is the upstream FaceForensics paper claim for EfficientNet on their test set, not our measured number on ours. We deliberately separate published claims from measured numbers throughout the report.

Q15Could you show us the fusion code?

Yes ma'am. The path is src forward slash modules forward slash fusion forward slash trust underscore scorer dot py. The function is compute underscore trust underscore score, on lines approximately 40 to 90. The pseudocode on slide 13 maps almost line by line to the actual implementation.

Q16Data privacy. Do you store the uploaded images and bios? Are there GDPR concerns?

No persistent storage ma'am. Images are written to a temporary file for inference and deleted immediately after. Bios are not stored at all. We log a request identifier and the processing time but no personally identifiable information. For a production deployment serving European users we would require a privacy policy and an explicit retention policy. The architecture is already aligned with the principle of data minimisation.
