03problem
Problem statement
You speak1:00

This is the problem we set out to address.

AI tools today can generate fake profile photos and fake bios that are indistinguishable from real ones to the human eye. Diffusion models such as Stable Diffusion and Midjourney generate the photos. Large language models such as ChatGPT generate the bios. Traditional checks like captchas and phone verification were never designed for this kind of threat.

The three numbers on the slide summarise the issue. It takes near zero seconds and near zero cost to generate a photo-realistic face online. There are public websites that produce a new synthetic person on every page refresh.

An attacker can fabricate three different signals independently. The image. The text. The metadata. Most platforms today still verify only one of these signals and consider the profile authenticated.

We observed this issue in everyday contexts. People share screenshots from social media without confirming that the photo is genuine. Recruiters receive fabricated LinkedIn profiles. Dating platforms encounter synthetic accounts. We wanted to build a tool that checks all three signals together and one that explains its reasoning so that a non-technical reviewer can understand why a profile has been flagged.

14stack
Stack · frontend half
You speak first0:35

For the frontend, we used React 18 with Vite as the build tool. Tailwind CSS for styling. SVG primitives for the gauges and the heatmap visuals. Playwright for end to end testing.

A brief justification for these choices. We selected Vite primarily for its development server speed. The sub-second hot reload was important given our timeline. Create React App would have been considerably slower in comparison.

Tailwind CSS helped us maintain consistent colours and spacing through its predefined design tokens. This kept the visual language coherent without the overhead of maintaining a separate style guide.

We chose SVG over a charting library deliberately. A library such as Recharts or Chart.js would have added between 30 and 50 kilobytes to the bundle. The trust gauge consists of a circle, a needle and four colored arcs, which SVG handles natively.

15api · ui
API and UI walkthrough · the live interface
You walk through1:50

This is the live interface of the application.

The user uploads a profile photo and pastes the bio text. On clicking Analyze, three categories of information are returned from the backend. The overall trust score is displayed prominently in the centre. The per module scores appear around it. The image score on one side. The text score on the other. The metadata position is shown only when the uploaded image carries EXIF data, since most social media uploads have their EXIF stripped.

Below these are the explanation panels. The Grad-CAM heatmap is displayed on the image side. The LIME token highlights appear on the text side. The colours map directly to importance. Red indicates that the model has identified that pixel or that word as pushing the verdict toward fake.

I would like to highlight three design decisions.

First. The trust band logic is implemented in a single helper file. In an earlier version, the same switch statement was duplicated across three components. The gauge, the verdict text and the explainability header. During one of our faculty reviews, the three components disagreed with each other. The gauge displayed low while the verdict text said very low. Consolidating that logic into one helper file eliminated this entire category of bug.

Second. The connection status banner polls the backend every five seconds and immediately when the user returns to the tab. If the backend becomes unavailable or comes back online, the user is informed within five seconds. The user is never left looking at a spinner without context.

Third. Tailwind purges any class that is not present in the source code at build time. So a colour class constructed from a runtime variable is removed from the production CSS. We encountered this issue during our first deployment. The solution is to list every possible colour class explicitly in a constant so Tailwind retains all of them. This pattern is necessary for any runtime driven styling.

§ 2 · Q&A · panel questionsPrepared answers.

Q01You did not use a state management library such as Redux or Zustand. As the application grows, will this approach become difficult to maintain?

For the current scope, introducing such a library would have added complexity that the project did not require. The application has a single screen, three top-level pieces of state and no cross-route communication. React's built-in useState and useReducer hooks handle this cleanly. If we add a history view or multi-page navigation in the future, I would introduce Zustand because it is lighter than Redux. Redux would only be appropriate if we required middleware or time travel debugging.

Q02The image upload supports files up to 10 megabytes. What is the user experience on a slow network connection and where is the progress feedback?

This is a known limitation in the current version, ma'am. We use the native fetch API with multipart form data, which does not expose upload progress directly. On a slow connection, the user sees the spinner but no percentage indicator. To provide real progress feedback, we would need to use XMLHttpRequest or the Streams API. In a production version, I would implement this and display a percentage bar.

Q03Polling the /health endpoint every five seconds. At scale, this results in hundreds of requests per second simply for status checks. Did you consider Server-Sent Events or WebSockets?

Yes ma'am. Polling was a deliberate choice for version 1.0 because the deployment was single instance and we wanted to avoid maintaining server-side state. For a multi-user production deployment, Server-Sent Events would be the appropriate approach. A single persistent connection per client, with the server pushing periodic heartbeats. WebSockets would also work but they add bidirectional capability that the application does not require, since traffic flows only from server to client.

Q04The trust band thresholds 0.30, 0.50 and 0.70 are defined in the frontend helper. If the backend team modifies a threshold and forgets to update the frontend, the application will be inconsistent.

This is a valid concern. The application would not crash but the gauge label and the verdict text would become incorrect. The proper solution is for the backend to return the band label and colour in the API response, so the frontend simply renders the values it receives. I had planned this refactor but did not include it in version 1.0. It is on my immediate priority list.

Q05Your palette is heavy on red. How does the trust gauge read for a viewer with red-green colour blindness?

That is a valid concern. Red and dark red on the same scale could merge for a deuteranope. We mitigated this by maintaining a clear shape difference between the four bands and by displaying the numeric score in the centre. Even if the colours merge for a particular viewer, the position on the dial and the numeric readout still convey the verdict. Strictly speaking, I should run the gauge through a colour blindness simulator such as Coblis to verify, which I have not done yet.

Q06The Grad-CAM heatmap is returned as a base64 encoded PNG inside the JSON response. Why not return a URL and let the browser fetch the image separately?

We chose the inline approach for two reasons. First, simplicity. A single round trip means less state to coordinate in the user interface. Second, the heatmap is single use and tightly bound to one analysis result, so caching it under a URL would have required additional logic for expiry. The trade-off is that each response is approximately 200 kilobytes larger. This was acceptable for version 1.0. For a high traffic deployment, I would switch to a URL based scheme.

Q07Race condition. The user uploads image A, does not wait and then uploads image B before the response for A returns. Which response is displayed by the user interface?

The most recent response is displayed. We track an active request identifier on the client and discard any response whose identifier does not match the current one. So if A's response returns after B has been submitted, A's result is dropped. This prevents the more serious failure mode of displaying stale results.

Q08Did you write any frontend unit tests, or only the end to end Playwright test?

Primarily Playwright, ma'am. We wrote a smoke test that exercises the full upload to verdict flow. We did not write component level unit tests for the trust band helper or the form validator. This is a gap in our test coverage. If I were starting the project again, I would write Vitest tests for those two helpers because they are pure functions and are straightforward to test.

Q09You selected Vite over Webpack. As Vite's plugin ecosystem matures over the next year, do you anticipate regretting this choice?

Vite was the correct choice for this project. The development server iteration speed was important for our timeline. For a long lived production application, the calculation might be different. For a final year project, the faster build cycle returned significant time over the course of development.

Q10The trust gauge uses a pie chart layout showing band proportions rather than a horizontal bar plotting the actual score, which would be more precise. Why did you choose the pie layout?

The pie chart communicates the band ranges, which is the information the user needs to interpret the verdict. A horizontal bar would communicate the score's position on the scale. They convey different things. We chose the pie chart for the main visualisation because the band geometry is what determines the verdict. A horizontal bar is also present on slide 12 as the smaller Figure 2 strip, so both views are available to the reviewer.
